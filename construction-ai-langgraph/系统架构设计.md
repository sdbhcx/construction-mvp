#### 3.3.2 智能体架构 (LangGraph + 决策回路)

现有流程是固定管道式调用。改为“感知-推理-行动-反思”的智能体决策回路，允许动态规划、工具选择、预算控制与人机协作。

**核心思想**
1. **状态驱动**：一切进入统一状态机(State)；节点只读写状态。
2. **规划先行**：Planner根据目标、上下文、工具成本生成计划；Executor执行；Critic评估后可调整。
3. **动态路由**：不是固定流程，基于意图、置信度、成本与风险选择路径。
4. **记忆体系**：短期对话记忆 + 任务记忆 + 长期向量记忆 + 结构化知识。
5. **护栏与审计**：策略约束、数据脱敏、执行日志、可追溯链路。

**智能体角色分层**
- **Coordinator Agent**：总体调度与预算控制，决定是否调用子智能体或人工复核。
- **Planner Agent**：将目标拆解为可执行步骤(可并行)，带工具与成本估计。
- **Extractor Agent**：多模态抽取，必要时迭代澄清。
- **Query Agent**：SQL/向量/知识库混合检索，带来源可信度。
- **Reviewer Agent**：质量评估、冲突检测、输出验收。
- **Tool Router**：统一工具入口，带权限与速率限制。

**系统级组件**
- **State Store**：统一状态(任务、置信度、证据、错误、预算)。
- **Memory Store**：短期缓存 + 长期向量 + 结构化知识库。
- **Policy Engine**：敏感字段处理、模型选择、费用上限、风险阈值。
- **Telemetry**：耗时、成本、召回率、覆盖率、复核率。

**决策回路示意 (Mermaid)**
```mermaid
flowchart TD
  A[Input] --> B[State Build]
  B --> C[Planner Agent]
  C --> D{Policy Check}
  D -->|Pass| E[Executor Agent]
  D -->|Reject| R[Fallback/Manual Review]
  E --> F[Tool Router]
  F --> G[OCR/VLM/NER/DB/Vector/KB]
  G --> H[State Update]
  H --> I[Critic Agent]
  I -->|Need Replan| C
  I -->|OK| O[Output + Audit]
```

**统一状态结构 (示例)**
```python
class TaskState(BaseModel):
  task_id: str
  goal: str
  inputs: Dict
  plan: List[Dict]
  evidence: List[Dict]
  confidence: Dict[str, float]
  cost: Dict[str, float]
  risk_flags: List[str]
  memory_refs: List[str]
  output: Dict
  status: str  # pending/running/review/complete/failed
```

**核心编排骨架**
```python
class AgentRuntime:
  def __init__(self, policy_engine, tool_router, memory_store):
    self.policy_engine = policy_engine
    self.tool_router = tool_router
    self.memory_store = memory_store
    self.planner = PlannerAgent()
    self.executor = ExecutorAgent()
    self.critic = CriticAgent()

  async def run(self, state: TaskState):
    while state.status not in {"complete", "failed"}:
      state.plan = await self.planner.plan(state)
      if not self.policy_engine.allow(state):
        state.status = "review"
        break

      state = await self.executor.execute(state, self.tool_router)
      verdict = await self.critic.evaluate(state)

      if verdict.replan:
        continue

      state.status = "complete"
    return state
```

**场景一: 文档抽取 (动态路径)**
1. Coordinator创建任务状态，加载历史记忆与项目规则。
2. Planner根据文档类型决定: OCR->NER->VLM 或 VLM->结构校验->补充OCR。
3. Executor按计划调用工具；Tool Router进行限流与模型选择。
4. Critic评估字段一致性与置信度，低于阈值进入复核。
5. 输出入库，并写回长期记忆与质量指标。

**场景二: 智能查询 (多工具组合)**
1. Planner选择SQL、向量、知识库三种检索策略(可并行)。
2. Executor执行并收集证据；Critic评估冲突并做答案合成。
3. 若证据不足，触发二次检索或澄清式追问。

**为什么更像真正Agent**
- 动态计划与反馈驱动的循环，而非固定管线。
- 统一状态与记忆驱动的长期改进。
- 策略与预算控制避免“盲目调用模型”。
- 可观测、可解释、可审计，满足工程落地。

## 4. 现有代码重构映射

为了将现有固定流程代码 (`construction-ai-langgraph`) 迁移到新的智能体架构，需要进行以下模块重构：

### 4.1 状态定义重构 (`src/schemas/state.py`)

现有的 `ConstructionState` 是扁平的流水线状态。需要转变为支持规划与元数据的分层状态。

```python
# 旧结构
class ConstructionState(TypedDict):
    ocr_results: Dict
    ner_results: Dict
    ...

# 新结构 (映射方案)
class TaskState(TypedDict):
    # 1. 任务上下文
    task: TaskInput
    
    # 2. 动态规划 (Planner生成)
    plan: List[PlanStep]       # [Step(tool="ocr"), Step(tool="vlm")]
    current_step_index: int
    
    # 3. 执行证据 (Executor收集)
    scratchpad: List[ToolMessage]  # 工具调用历史
    observations: Dict[str, Any]   # 结构化观测结果 { "ocr_text": "...", "tables": [...] }
    
    # 4. 评估与反思 (Critic生成)
    critique: Optional[CritiqueResult] # { "approved": False, "reason": "Missing date" }
    retry_count: int
```

### 4.2 节点转工具化 (`src/nodes/` -> `src/tools/`)

目前的 `nodes` 是图中的硬编码节点。需要将其封装为标准工具，供 Planner 调度。

| 现有节点 (Node) | 重构目标 (Tool) | 描述 |
| :--- | :--- | :--- |
| `run_ocr_node` | `OCRTool` | 输入图像，输出原始文本与坐标。独立封装 PaddleOCR 调用。 |
| `detect_tables_node` | `TableExtractionTool` | 专门处理表格区域，返回 CSV/Markdown 格式数据。 |
| `run_vlm_node` | `VLMAnalysisTool` | 接受 Prompt 和图像，返回结构化 JSON。 |
| `run_ner_node` | `NERTool` | 纯文本实体抽取工具。 |
| `sql_query_nodes` | `DatabaseTool` | 安全的 SQL 执行沙箱。 |

### 4.3 编排逻辑迁移 (`src/graphs/` -> `src/agents/`)

将硬编码的 `StateGraph` 转变为智能体类。

*   **删除** `ConstructionGraph` (固定流水线)。
*   **新增** `PlannerAgent` (`src/agents/planner.py`):
    *   职责：接收 `TaskState`，分析 `file_type`，生成步骤列表。
    *   示例逻辑：`Pre-condition: file_type='pdf' -> Plan: [OCR, LayoutAnalysis, Extraction]`
*   **新增** `ExecutorAgent` (`src/agents/executor.py`):
    *   职责：遍历 `plan`，调用 `ToolRouter` 执行对应工具，更新 `observations`。
*   **新增** `ReviewAgent` (`src/agents/reviewer.py`):
    *   职责：替代原有的 `validate_extraction_node`。不但检查格式，还根据业务规则（如“金额必须匹配”）决定是否 `replan`。

### 4.4 目录结构调整建议

建议将目录结构调整为以“能力”为核心，而非“技术层”为核心：

```text
src/

  │   ├── state.py          # 统一状态定义
  │   ├── memory.py         # 记忆管理 (Redis/VectorDB)
  │   └── policy.py         # 策略控制 (预算、权限)
  ├── capabilities/         # 原子能力 (原 nodes)
  │   ├── vision/           # OCR, VLM
  │   ├── text/             # NER, Extraction
  │   └── storage/          # DB, Vector
  ├── agents/               # 智能体 (大脑)

  │   ├── executor.py       # 执行者
```

### 4.5 迁移步骤

1.  **Phase 1 - 工具化**: 将 `ocr_nodes.py` 等重构为 `src/tools/ocr_tool.py`，确保其输入输出标准化。
2.  **Phase 2 - 状态升级**: 定义新的 `TaskState`，并修改 `load_document` 节点以初始化新状态。
3.  **Phase 3 - 引入 Loop**: 创建 `AgentRuntime` 类，实现 `Plan -> Execute -> Critic` 的 while 循环。
4.  **Phase 4 - 逐步替换**: 在 Loop 中首先只调用 `Executor` 执行旧的固定流程，验证通过后，逐步开启 `Planner` 的动态生成能力。

### 5.2 策略引擎 (Policy Engine)

负责在执行每一动作前进行合规与预算检查。

**核心逻辑 (`src/core/policy.py`)**

```python
class PolicyEngine:
    def __init__(self, config):
        self.max_steps = 20
        self.budget_limit = 5.0 # USD
        self.sensitive_keywords = ["身份证", "手机号"]

    def check_action(self, state: TaskState, action: ToolAction) -> bool:
        # 1. 预算检查
        if state.cost_acc + action.estimated_cost > self.budget_limit:
            raise PolicyViolation("Budget Exceeded")
            
        # 2. 循环检测
        if len(state.plan) > self.max_steps:
             raise PolicyViolation("Max Steps Reached")

        return True
    
    def sanitize_output(self, content: str) -> str:
        """脱敏处理"""
        for kw in self.sensitive_keywords:
            content = content.replace(kw, "***")
        return content
```

### 5.4 智能体详细设计

#### 5.4.1 Planner Agent (规划者)

负责将高层目标拆解为具体的工具执行序列（DAG）。

*   **职责**:
    *   分析任务输入（文件类型、用户指令）。
    *   检索类似任务的成功计划（Few-shot）。
    *   生成包含依赖关系的步骤列表。
    *   动态调整计划（当 Executor 反馈失败或 Critic 驳回时）。

*   **Prompt 策略**: `ReAct` 或 `Plan-and-Solve` 变体。
*   **输出结构**:
    ```python
    class PlanStep(BaseModel):
        id: str
        tool_name: str
        tool_input: Dict
        reason: str
        dependencies: List[str]  # 依赖的前序步骤ID
    
    class ExecutionPlan(BaseModel):
        steps: List[PlanStep]
        estimated_cost: float
    ```
*   **核心逻辑 (`src/agents/planner.py`)**:
    ```python
    async def create_plan(self, state: TaskState) -> ExecutionPlan:
        # 1. 检索上下文 (Memory)
        examples = await self.memory.retrieve_relevant_plans(state.goal)
        
        # 2. 调用 LLM 生成计划
        system_prompt = f"""
        你是施工文档处理专家。请根据目标生成执行步骤。
        可用工具: {self.tools_description}
        参考案例: {examples}
        """
        response = await self.llm.generate(system_prompt, state.inputs)
        return parse_plan(response)
    ```

#### 5.4.2 Executor Agent (执行者)

负责具体的工具调用与错误恢复，不进行复杂的规划决策。

*   **职责**:
    *   读取 Plan，解析当前可执行的步骤（解决依赖）。
    *   通过 `PolicyEngine` 校验工具调用参数。
    *   执行工具（Tool Router -> 实际 Docker 服务）。
    *   标准化工具输出（Observation）。
    *   处理工具级错误（如重试、参数修正）。

*   **执行循环**:
    ```python
    async def execute_step(self, step: PlanStep, state: TaskState):
        try:
            # 1. 准备输入 (从之前步骤的输出中提取参数)
            inputs = self._resolve_inputs(step.tool_input, state.observations)
            
            # 2. 策略检查
            self.policy.check(step.tool_name, inputs)
            
            # 3. 调用工具
            result = await self.tool_router.call(step.tool_name, inputs)
            
            # 4. 记录观测
            state.observations[step.id] = result
            state.scratchpad.append(ToolMessage(step.id, result))
            
        except ToolError as e:
            # 尝试自我修正参数或标记步骤失败
            if self._can_retry(e):
                return await self.execute_step(step, state)
            state.error = str(e)
    ```

#### 5.4.3 Critic Agent (审查者/反思者)

负责质量把控与结果验收，是闭环的关键。

*   **职责**:
    *   **一致性检查**: 比如“OCR识别的日期”与“用户要求的日期”是否匹配。
    *   **完整性检查**: 所有必填字段是否已从文档中提取。
    *   **置信度评估**: 综合各字段的置信度，决定是否通过。
    *   **决策输出**: `Pass` (完成) / `Replan` (需补充步骤) / `HumanReview` (转人工)。

*   **Prompt 策略**: Chain-of-Thought (CoT) + Rule-based Validation
*   **评估逻辑 (`src/agents/critic.py`)**:
    ```python
    async def evaluate(self, state: TaskState) -> Verdict:
        # 1. 硬规则校验 (Rule Engine)
        if missing := self.validator.check_required_fields(state.output):
            return Verdict(action="replan", reason=f"缺失字段: {missing}", adjustments=["增加OCR区域"])
            
        # 2. 语义一致性校验 (LLM)
        evaluation = await self.llm.evaluate(
            xml_content=state.observations.get("xml"),
            user_instruction=state.task.goal
        )
        
        if evaluation.score < 0.8:
            return Verdict(action="human_review", reason="置信度过低")
            
        return Verdict(action="complete")
    ```

```python
# Redis缓存策略
class CacheManager:
    """缓存管理"""
  
    def __init__(self):
        self.redis = redis.Redis(
            host=config.REDIS_HOST,
            port=config.REDIS_PORT,
            db=0,
            decode_responses=True
        )
  
    # 缓存键设计
    CACHE_KEYS = {
        # 用户相关
        'user:{user_id}': {'ttl': 3600, 'desc': '用户信息'},
      
        # 文档相关
        'document:{doc_id}': {'ttl': 1800, 'desc': '文档信息'},
        'document_status:{doc_id}': {'ttl': 300, 'desc': '文档状态'},
      
        # AI结果缓存
        'ocr_result:{file_hash}': {'ttl': 86400, 'desc': 'OCR结果'},
        'vlm_result:{file_hash}': {'ttl': 86400, 'desc': 'VLM结果'},
        'query_result:{query_hash}': {'ttl': 3600, 'desc': '查询结果'},
      
        # 配置缓存
        'project_config:{project_id}': {'ttl': 3600, 'desc': '项目配置'},
        'model_config:{model_type}': {'ttl': 1800, 'desc': '模型配置'},
      
        # 会话缓存
        'session:{session_id}': {'ttl': 7200, 'desc': '用户会话'},
      
        # 限流缓存
        'rate_limit:{user_id}:{action}': {'ttl': 60, 'desc': '限流计数'}
    }
  
    async def get_with_cache(self, key, func, *args, **kwargs):
        """带缓存的获取"""
        # 先查缓存
        cached = await self.redis.get(key)
        if cached:
            return json.loads(cached)
      
        # 缓存未命中，执行函数
        result = await func(*args, **kwargs)
      
        # 设置缓存
        ttl = self.CACHE_KEYS.get(key.split(':')[0] + ':*', {}).get('ttl', 300)
        await self.redis.setex(key, ttl, json.dumps(result))
      
        return result
```

```python
# JWT认证中间件
class JWTAuthMiddleware:
    """JWT认证中间件"""
  
    def __init__(self, secret_key: str, algorithm: str = "HS256"):
        self.secret_key = secret_key
        self.algorithm = algorithm
      
    async def __call__(self, request: Request, call_next):
        # 跳过公开路径
        if request.url.path in ["/api/v1/auth/login", "/health", "/docs"]:
            return await call_next(request)
      
        # 获取Token
        auth_header = request.headers.get("Authorization")
        if not auth_header or not auth_header.startswith("Bearer "):
            raise HTTPException(status_code=401, detail="未提供认证Token")
      
        token = auth_header.split(" ")[1]
      
        try:
            # 验证Token
            payload = jwt.decode(
                token, 
                self.secret_key, 
                algorithms=[self.algorithm]
            )
          
            # 检查权限
            if not self._check_permission(payload, request):
                raise HTTPException(status_code=403, detail="权限不足")
          
            # 将用户信息添加到请求
            request.state.user = payload
          
        except jwt.ExpiredSignatureError:
            raise HTTPException(status_code=401, detail="Token已过期")
        except jwt.InvalidTokenError:
            raise HTTPException(status_code=401, detail="无效Token")
      
        return await call_next(request)
  
    def _check_permission(self, payload: Dict, request: Request) -> bool:
        """检查权限"""
        user_role = payload.get("role", "viewer")
        required_role = self._get_required_role(request)
      
        # 角色权限映射
        role_hierarchy = {
            "admin": ["admin", "manager", "engineer", "viewer"],
            "manager": ["manager", "engineer", "viewer"],
            "engineer": ["engineer", "viewer"],
            "viewer": ["viewer"]
        }
      
        return required_role in role_hierarchy.get(user_role, [])

# RBAC权限控制
class RBACPermission:
    """RBAC权限控制"""
```
